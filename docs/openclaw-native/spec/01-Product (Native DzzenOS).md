# DzzenOS (OpenClaw Native) — продукт

## 0) Коротко: что мы строим
**DzzenOS** в нативной версии — это:
- Канбан‑доски + документы + агентные сессии **внутри OpenClaw**
- Карточка задачи = *контейнер контекста* + *чат‑сессия агента* + *артефакты* + *апрувы*
- Всё исполняется **локально** (на железе пользователя), используя его модели/провайдеры

Цель: сделать “операционку” для соло‑фаундера (Founder Ops + Content) — быстро, умно, приватно.

---

## 1) Основные сущности (в терминах UX)

### 1.1 Workspace (пространство)
- отдельный проект/компания/направление
- содержит несколько досок
- содержит общие документы (память)

### 1.2 Board (доска)
- доска — это не просто список задач, а **контейнер политики**:
  - дефолтный агент/профиль
  - режим работы (быстро/качественно/дешево)
  - ограничения и правила (апрувы, лимиты, запрет внешних действий)

### 1.3 Task Card (карточка)
Карточка — главный объект.
В карточке обязательно:
- **Title + Brief** (входные данные)
- **Agent Session** (чат/контекст)
- **Runs** (история запусков)
- **Artifacts** (результаты)
- **Approvals** (если требуется)

Смысл: пользователь общается с агентом “внутри карточки”, уточняет вводные, и запускает выполнение.

### 1.4 Docs / Memory
Документы бывают:
- workspace‑level (общая память проекта)
- board‑level (контекст под домен: Content, Founder Ops)
- task‑level (локальный контекст задачи — чаще это просто brief + вложения)

---

## 2) MVP доски: Founder Ops и Content

### 2.1 Founder Ops (MVP)
**Смысл:** входящие → план → выполнение → weekly review.

Рекомендуемые статусы (единые):
- Inbox → Backlog → Next → In Progress → Review → Blocked → Done

Примеры карточек:
- weekly review
- план недели
- список тем контента
- план экспериментов

### 2.2 Content board (MVP)
Поверх общих статусов делаем view/фильтры “как контент‑пайплайн”, а не отдельные статусы.
Например view «Content pipeline»:
- Inbox (Ideas)
- Next (Draft)
- In Progress (Write)
- Review (Edit)
- Done (Published)

Артефакты:
- `draft` (черновик)
- `article_md`
- `social_pack`

---

## 3) Как работает агент в карточке

### 3.1 Встроенная сессия
У каждой карточки есть своя “сессия агента”:
- отдельная история сообщений
- отдельный набор attached docs (контекст)
- отдельные настройки (override политики board)

### 3.2 Паттерн работы
1) пользователь заводит карточку (Brief)
2) агент задаёт 2–5 уточняющих вопросов
3) пользователь нажимает Start
4) агент выполняет шаги, пишет прогресс, создаёт артефакты
5) если нужно действие “наружу” → approval

---

## 4) Базовое решение бесплатным — да
Стратегия “free core” логична:
- бесплатный: boards/tasks/docs/agent sessions/runs
- платное потом: пакеты шаблонов, премиальные agent packs, синхронизация, командный режим, бренд‑пакеты и т.д.

Важно: монетизация не должна ломать основной принцип: **локально и приватно**.
